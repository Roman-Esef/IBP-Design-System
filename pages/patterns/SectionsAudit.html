Предложение: единый шаблон страницы

Так эту задачу решают зрелые дизайн-системы: у Carbon (IBM) каждая страница компонента — это фиксированные вкладки Usage / Style / Code / Accessibility, у Material — Overview / Specs / Guidelines / Accessibility. Набор разделов жёстко одинаковый, а всё уникальное живёт внутри разделов как подзаголовки. Уникальные разделы не удаляются — они понижаются до h3 внутри канонического h2.

Канонический набор разделов (фиксированный порядок)

  1. Конструктор — интерактивный playground
  2. Использование — когда использовать / когда не использовать; сравнение с соседними компонентами («SegmentControl vs Tab») — подраздел здесь
  3. Анатомия — схема с маркерами
  4. Варианты — все оси конфигурации: типы, тоны, формы, ориентация, режимы («С бейджем», «Встроенный вариант», «Split Button», toggle…) — каждый как h3 новый раздел
  5. Размеры — размерный ряд и redline-схемы
  6. Контент — правила текста, иконографики, переполнение текста
  7. Поведение — позиционирование, авто-разворот, стек, анимация, тайминги, адаптивность новый раздел, опционален
  8. Состояния — Default / Hover / Pressed / Disabled / Error…
  9. Доступность — ARIA, клавиатура, область клика
  10. Типографика — используемые токены --type-*
  11. Цвета — используемые семантические токены
  12. Для разработчиков — хендофф-пакет

Правила шаблона

  • Имена h2 — контракт. Только слова из списка выше, без вариаций. Бейджи («Новое», «Расширение») — отдельным элементом внутри h2, не частью текста заголовка.
  • Обязательные разделы присутствуют всегда. Если раздел неприменим (например, «Типографика» у IconButton — текста нет), он всё равно существует и содержит одну строку: «Не применимо: …». Агент никогда не получает «раздел не найден».
  • «Варианты» и «Поведение» — контейнеры. Уникальные разделы конкретного компонента становятся их h3-подразделами. Список h3 свободный — контракт только на уровне h2.
  • Синонимы переименовываются, контент не трогаем: «Когда использовать» → «Использование», «Текстовка» → «Контент», «Доступность и область клика» → «Доступность» (область клика — h3 внутри).
  • Спеки повторяют шаблон. В specs/<Имя>.md — те же 12 разделов в том же порядке; агент-сборщик читает спеку, а не страницу.
  • «Что предлагаем добавить» / «Предложение» — не раздел документации, а рабочий бэклог. Выносится в конец страницы отдельным блоком вне контракта (или в общий бэклог-файл).

Порядок работ

  • 1. Утвердить набор и имена разделов (этот документ).
  • 2. Обновить CLAUDE.md и specs/_TEMPLATE.md — зафиксировать контракт.
  • 3. Механическая волна: переименовать синонимы h2 на всех страницах (без переноса контента) — матрица выше станет почти зелёной.
  • 4. Структурная волна, по компоненту за раз: собрать «Варианты» и «Поведение», понизить уникальные разделы до h3, синхронизировать спеки.
  • 5. Добавить недостающие обязательные разделы (в т.ч. «Не применимо»).
