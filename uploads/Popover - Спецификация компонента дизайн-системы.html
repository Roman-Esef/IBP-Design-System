Обзор и место в иерархии

Popover — нон-модальный всплывающий контейнер, привязанный к триггеру

Popover показывает расширенный контекстный контент: текст с форматированием, интерактивные элементы (ссылки, кнопки), формы, легенды, списки. В отличие от tooltip — требует клика для открытия и не закрывается само по себе при уходе курсора.

01

Анатомия компонента

Зоны, обязательность, ARIA-роли. Компонент состоит из 2 обязательных и 5 опциональных частей

  • 1

    Trigger Обязательно

    Любой интерактивный элемент: кнопка, иконка, текстовая ссылка, badge. Получает aria-expanded и aria-controls. Управляет видимым focus ring.

  • 2

    Container Обязательно

    Обёртка с тенью и скруглёнными углами. Несёт role="dialog" или role="region", aria-modal="false" (не блокирует страницу).

  • 3

    Caret / Arrow Опционально

    Визуальная стрелка-указатель на триггер. Помогает при нескольких соседних триггерах. Можно скрыть для контекстных меню и detached-поповеров.

  • 4

    Header Опционально

    Содержит только заголовок и кнопку ✕. Добавляется, когда контент требует контекста или при ширине свыше 320 px. Лимит заголовка — 25 символов в одну строку. Кнопки действий в Header не размещаются — для них предназначен Footer.

  • 5

    Body / Content area Обязательно

    Слот под произвольный контент: текст, списки, формы, чарты, таблицы. Единственная обязательная внутренняя зона. Прокручивается при overflow.

  • 6

    Footer Опционально

    Зона кнопок действий и пагинации. Используется при форме или подтверждении внутри поповера. Все интерактивные действия (кроме закрытия) живут именно здесь.

  • 7

    Close button (✕) Рекомендовано

    aria-label="Закрыть". Размещается в правом верхнем углу Header (или Body — если Header нет). Обязательна для a11y и предсказуемости.

3
Можно ли размещать кнопку действия в Header? Нет. Header содержит только заголовок и кнопку закрытия (✕). Любые действия — «Сохранить», «Применить», «Отмена», «Открыть сделку», «Подробнее» — размещаются только в Footer. Это правило обеспечивает предсказуемость взаимодействия и однозначное определение зоны действий для пользователя и скринридера. Исключение — служебные иконки-утилиты в Header (например, «Сбросить фильтры» рядом с ✕), оформленные как ссылки, а не как primary-кнопки.

1
Какое ограничение по высоте Popover? Высота определяется контентом (auto height) и ограничена сверху значением max-height = 60vh или 400–520 px в зависимости от ширины (см. таблицу выше). При превышении max-height появляется вертикальная прокрутка только внутри Body — Header и Footer остаются зафиксированными. Горизонтальная прокрутка запрещена. Если контент не помещается даже в xl-размер с прокруткой — это сигнал переключиться на Modal или Drawer.

Алгоритм auto-flip (Floating UI / Popper.js)

Preferred position → detectOverflow() → flip() → противоположная сторона → shift() → ближайший край

При каждом из 4 направлений возможны три варианта выравнивания (start, center, end) — это позволяет избегать перекрытия важного контента без смены стороны открытия. Edge detection: поповер не выходит за границы экрана; при необходимости смещается по оси start/end. Минимальный отступ от края viewport — 8 px.

Анимация enter / leave

Параметры переходов и поведение при prefers-reduced-motion

ПараметрЗначение
Enter duration150–180 ms
Leave duration100–120 ms
Easing (enter)cubic-bezier(.16,1,.3,1)
Easing (leave)cubic-bezier(.4,0,1,1)
Initial transformtranslateY(-4px) scale(.98)
Transform originот триггера (авто, рассчитывается из placement)
Initial opacity0
ℹ
prefers-reduced-motion: при включённой системной настройке анимация отключается, поповер появляется и скрывается мгновенно. Это требование WCAG 2.3.3.

CSS-пример

.popover {
  opacity: 0;
  transform: translateY(-4px) scale(.98);
  transition: opacity 180ms cubic-bezier(.16,1,.3,1),
              transform 180ms cubic-bezier(.16,1,.3,1);
}
.popover.open {
  opacity: 1;
  transform: translateY(0) scale(1);
}
@media (prefers-reduced-motion: reduce) {
  .popover { transition: none; }
}

Контент однозначен и самодостаточенНе нужен

Форма с несколькими полямиДобавить

Лимит заголовка: не более 25 символов, в одну строку (Red Hat). Header содержит только заголовок и кнопку ✕; primary-кнопки действий в Header не размещаются.

Footer — когда добавлять?

Только информационный контентНе нужен
Внутри есть форма или выбор, требующий подтвержденияДобавить
Деструктивное действие, требующее confirmДобавить
Ссылка «Подробнее» / внешний переходМожно в Footer или в конец Body
Все интерактивные действия (Сохранить, Применить, Отмена, Открыть, Удалить) живут в Footer. Это снимает неоднозначность для пользователя и скринридера.

Порядок и выравнивание кнопок в Footer

КОНСТРУКТИВНЫЕ ДЕЙСТВИЯ — СПРАВА
Footer:

ИНФО / СЛУЖЕБНЫЕ ССЫЛКИ — СЛЕВА
Footer: Найдено: 24

Информационные подписи («Найдено N», «Показано 5 из 12»), служебные ссылки («Сбросить», «Полный отчёт», «История») — слева. Они не конкурируют с основным действием за фокус внимания.

2
В каких случаях кнопки в Footer размещаются слева, а в каких — справа? Справа размещаются конструктивные / primary действия, завершающие задачу: «Применить», «Сохранить», «Подтвердить», «Открыть сделку». Рядом с ними слева идёт парная вторичная кнопка «Отмена» / «Назад». Этот порядок принят в Salesforce, Carbon, Material — он соответствует прогрессивному движению взгляда и финальной точке принятия решения. Слева размещаются служебные и информационные элементы: счётчики («Найдено N», «Показано 5 из 12»), кнопки-ссылки («Сбросить», «История оценок», «Полный отчёт», «Экспорт CSV»), а также деструктивные действия в confirm-поповерах (например, «Закрыть без сохранения») — чтобы пользователь не нажал их случайно при движении к primary-кнопке справа.

Нестандартные сценарии и граничные состояния

Async-загрузка контента

Loading state

Когда данные загружаются из API после открытия — показываем skeleton в Body. Не блокируем открытие. Header рендерится сразу. aria-busy="true" на body. Минимальная высота не прыгает.

Пустое состояние (empty)

Empty state

Никогда не показывать пустой контейнер. Иконка + человеческий текст + CTA-действие. Footer в этом кейсе убирается (или содержит только основной CTA).

Error state

API вернул ошибку — не закрываем поповер. Показываем error state с кнопкой повтора. role="alert". Никогда не показывать stack trace или HTTP-коды — только человеческое сообщение.

Доступность (WCAG 2.2 AA)

Обязательные атрибуты, фокус, клавиатурная навигация и контрастность

ARIA-атрибуты

✓
role="dialog" на контейнере

Обязателен при интерактивном контенте. При отсутствии интерактивных элементов — role="region" с aria-label. Не role="tooltip".

✓
aria-modal="false"

Поповер не блокирует основной контент. Скринридер сохраняет доступ к остальному.

✓
aria-expanded на триггере

false при закрытом, true при открытом. Меняется атомарно.

✓
aria-controls="[popover-id]"

Связывает триггер с конкретным поповером.

✓
aria-haspopup="dialog"

Тип всплывающего контента для скринридера.

✓
aria-labelledby на диалоге

Указывает на заголовок. Если заголовка нет — aria-label.

✓
Кнопка ✕ — aria-label="Закрыть"

Без видимого текста. Только иконка + a11y-метка.

Клавиатура и фокус

КлавишаДействие
Enter / SpaceОткрыть/закрыть на триггере
TabПереход между интерактивными элементами внутри
Shift+TabОбратный обход

✓
Открытие — фокус на первый интерактивный

Внутри поповера. Для нон-модального soft trap — Tab может уйти за пределы и закрыть.

✓
Закрытие — возврат фокуса на триггер

Точно на тот элемент, который открыл поповер. Требование WCAG 2.4.3.

✓
Контрастность текста

≥ 4.5:1 для обычного текста, ≥ 3:1 для крупного (≥18 px). Граница / тень контейнера обеспечивает визуальное отделение от фона.

✓
Видимый focus ring

:focus-visible на триггере и интерактивных элементах внутри. Минимум 2 px outline.

Z-index стэкинг

Место поповера в стэке наложения относительно других overlay-компонентов

Toast1080

Popover1000–1060
Modal1050
Drawer1040
Sticky header100
Base content1

Поповер должен находиться выше Modal чтобы при необходимости появляться поверх открытого модального окна — редкий, но реальный кейс в enterprise-системах (например, поповер-подсказка по полю формы внутри модалки).

⚠
Никогда не открывайте поповер поверх другого поповера. Это нарушает фокус-модель и неоднозначно для Esc — какой поповер закрывать первым? Если нужен новый — закрывай старый.

Tooltip — выше поповера (1070): подсказка по элементу внутри поповера должна быть видна. Toast (1080) — поверх всего: критические нотификации не должны перекрываться поповером.

Do / Don't — правила применения

Что делать и чего избегать

✓ Делать

✓
Использовать для контекстного контента

Поповер раскрывает дополнительный контент, не требующий полной страницы.

✓
Фиксированная ширина из токенов

Всегда один из 5 размеров: xs/sm/md/lg/xl. Никогда не резиновая ширина.

✓
Возвращать фокус на триггер

После любого закрытия фокус идёт обратно на элемент, открывший поповер.

✓
Показывать шапку при сложном контенте

3+ смысловых блока, форма, несколько действий — заголовок обязателен. Header содержит только заголовок и ✕.

✓
Размещать действия только в Footer

Primary-кнопки — справа, вторичные и служебные ссылки — слева. Деструктивные действия в confirm — слева.

✓
Ограничить тело max-height с прокруткой

Body прокручивается, Header и Footer фиксированы. Горизонтальная прокрутка запрещена.

✓
Поддерживать Esc и outside click

Все 5 способов закрытия должны работать одновременно.

✓
Документировать паттерны отдельно

Popover/Легенда, Popover/Форма, Popover/Статус — отдельные шаблоны над базовым компонентом.

✗ Не делать

✗
Не превышать 560 px ширины

Шире 560 px → переключайтесь на Modal/Drawer/отдельную страницу.

✗
Не использовать только hover

Недоступно на тач-устройствах и для клавиатуры. Click — основной триггер.

✗
Не закрывать по таймауту

Пользователь не успевает прочитать. Только явное действие пользователя.

✗
Не открывать при загрузке страницы

Нарушает Heuristic №3 Nielsen (пользовательский контроль).

✗
Не размещать кнопки действия в Header

Header — только заголовок и ✕. Все действия живут в Footer (см. ответ на вопрос 3 в анатомии).

✗
Не вкладывать больше 1 уровня

Поповер в поповере: максимум 2 уровня, второй — только read-only xs.

✗
Не открывать несколько одновременно

Открытие нового автоматически закрывает предыдущий. Исключение: вложенный xs.

✗
Не делать горизонтальный скролл

Дезориентирует. Увеличить ширину или Modal.

✗
Не закрывать при Esc если есть unsaved changes

При dirty state — confirm-подтверждение «Несохранённые изменения — закрыть?».
