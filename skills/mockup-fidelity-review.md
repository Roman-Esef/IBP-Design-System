# Скилл: Приёмка экрана по макету (ревьюер · гейт фиделити)

**Роль:** въедливый ревьюер. Задача — **не одобрить, а найти расхождения** рендера с макетом. По замыслу враждебный: снисходительность здесь и провалила прошлый заход.
**Когда:** после сборки экрана (`skills/screen-from-mockup.md`), до `ready_for_verification`. Это отдельный гейт, а не пункт в билдере — чтобы его нельзя было «пробуквить».
**Вход:** макет (`view_image` референса) + рендер экрана. **Выход:** вердикт **PASS** / **NEEDS-WORK** + список конкретных дефектов.

---

## Протокол (обязательные шаги)
1. **Целевая ширина, БЕЗ временных override.** Открыть рендер на ширине макета и захватить то, что файл рендерит **сам** (никаких `!important`/отключения `@media` в скриншот-коде — пользователь этого не увидит). Мокап 1920 обязан держать десктоп-раскладку на любой ширине превью (пин `min-width` + горизонтальный скролл); если она держится только под override — это дефект файла (класс 7), а не приёмка. Скриншот в узком превью, где сработал адаптивный брейкпоинт со схлопыванием, вердикт делает невалидным — и сам такой брейкпоинт в мокапе есть FAIL.
2. **Масштаб 1:1 для типографики.** Хотя бы один регион снять без zoom (1:1): мелкие zoom-обзоры скрывают реальные кегли/отступы — именно на них прошлый заход ошибочно поставил PASS.
3. **Оверлей/рядом.** Держать `view_image` макета и скриншот рендера рядом. Идти **регион за регионом** (шапка → табы → тулбар → каждая карточка → таблица → сайдбар → модалка).
4. **Прогнать рубрику** по каждому региону. Любой FAIL хотя бы в одном классе → вердикт **NEEDS-WORK**.
5. **Вернуть дефекты адресно:** регион + класс дефекта + что в макете vs что в рендере.

## Рубрика (классы дефектов, критерии PASS)
| # | Класс | PASS, когда… | Типичные FAIL |
|---|---|---|---|
| 1 | **Плотность** | паддинги карточек, зазоры сеток, высоты строк совпадают с макетом (±небольшой допуск) | всё воздушнее макета, «прототипный» вид, крупные gap |
| 2 | **Типошкала** | относительные кегли и веса лейблов/значений/тайтлов как в макете | body-токены «как есть», лейблы/значения крупнее оригинала |
| 3 | **Выравнивание** | колонки, чипы, иконки, числа (right-align, tabular-nums) стоят как в макете | разъехавшиеся `grid`, чипы перенесены, числа влево |
| 4 | **Полнота** | присутствуют **все** регионы, строки, колонки и данные макета | пропущены колонки/строки, «сокращённые» блоки |
| 5 | **Отрисовка vs заглушка** | блоки макета отрисованы на CSS; заглушка только для внесмасштабного и подписана | allocation-бар/степпер/карточки заменены `.stub` без причины |
| 6 | **Цвет/токены** | цвета статусов, акценты, фоны — из токенов ДС и совпадают по смыслу с макетом | выдуманные цвета, неверный тон статус-пилюли |
| 7 | **Структура/компоненты** | использованы правильные классы ДС; DOM-сниппеты валидны (нет `.cbx`/`.toast__text` вместо `.cb`/`.toast__msg`) | классы по памяти, сломанная разметка компонента |
| 8 | **Иконки** | только `<i data-icon>`, глиф существует в `specs/Icons.md`, смысл совпадает | инлайн-SVG, несуществующий/неверный глиф |
| 9 | **Консоль/ДС** | нет ошибок в консоли; `check_design_system` чист | битые ссылки на стили/скрипты, невалидная карточка `@dsCard` |

## Форма вердикта
```
ЭКРАН: <файл>  ШИРИНА ПРОВЕРКИ: 1920
ВЕРДИКТ: PASS | NEEDS-WORK
Дефекты (если есть):
- [<класс>] <регион>: <макет> vs <рендер> → <что править>
Расхождения токенов (принятые): <если ближайший токен ≠ макет — зафиксировать здесь>
```

## Правила вердикта
- **NEEDS-WORK**, если есть хоть один FAIL классов 1–5 или 7 (это то, что рушит сходство/корректность). Классы 6, 8, 9 — тоже FAIL, но чаще точечные.
- **PASS** только когда все классы ок на целевой ширине. После PASS билдер зовёт `ready_for_verification`.
- Придирки ради придирок (микросдвиг в 1px, не влияющий на восприятие) — не FAIL. Гейт про **сходство и корректность**, а не про перфекционизм ради него самого.
- Не чинить самому в роли ревьюера — вернуть дефекты билдеру списком; правки и повторный прогон — его цикл (§4 билдера).
