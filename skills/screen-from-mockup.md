# Скилл: Экран по макету (билдер)

**Роль:** топовый дизайнер интерфейсов, собирающий экран из компонентов и токенов ДС **пиксельно близко к присланному макету**.
**Парный артефакт:** `skills/mockup-fidelity-review.md` — гейт приёмки. Экран нельзя отдавать (`ready_for_verification`), пока не прошёл его рубрику.
**Язык:** русский. **Формат ДС:** ванильный HTML/CSS, без React.

> Этот скилл написан по факту провала: 9 экранов собрали «на глаз» в узком превью, без сверки с макетом на 1920 → разъехалась плотность, поломались плитки, слишком много заглушек. Каждый шаг ниже закрывает конкретную причину.

---

## 0. Железные правила (нарушение = переделка)
1. **Сначала — готовый компонент ДС, потом своя вёрстка.** Каждый регион макета сперва ищем в ДС (`specs/_cheatsheet.md`) и собираем из её компонентов + токенов. Свой CSS — только раскладка (grid/flex/gap) и то, чего в ДС нет как компонента. Похоже на существующий компонент, но «чуть иначе» → используем компонент, а не рисуем двойник; расхождение — в отчёт гейта.
2. **Судить рендер только на ширине макета.** Никогда не оценивать экран в узком превью — там сетки схлопываются, и ты видишь не то, что отдашь. Форсить целевую ширину (см. §4).
3. **Скриншот-diff обязателен**, а не опционален. Это самый дешёвый способ не пересобирать ×N. Один лишний скриншот дешевле одного переделанного экрана.
4. **Плотность — раньше красоты.** Сначала попасть в кегли/паддинги/высоты строк макета, потом всё остальное.
5. **Рисуем, а не заглушаем.** Заглушка (`.stub`) — только для реально внесмасштабного (тяжёлый чарт-движок) и с явной подписью-пометкой. Allocation-бар, степпер, карточки, таблицы — рисуются на CSS.
6. **Только токены и классы ДС.** Ни одного выдуманного цвета/размера/радиуса/шрифта. Нет токена — спроси или возьми ближайший и пометь в отчёте.
7. **Экран 01 — эталон.** Довести первый до pass, показать пользователю, и только потом тиражировать плотность и приёмы на остальные. Даже если просили «собери все сразу» — собирай все, но каждый прогоняй через гейт до отдачи.
8. **Мокап = фиксированный десктоп-канвас.** Экран под 1920 всегда держит десктоп-раскладку: пин `min-width` на контент, узкое окно → горизонтальный скролл. НЕ вводить адаптивные `@media`-брейкпоинты со схлопыванием колонок: превью пользователя обычно уже ширины макета, и он увидит именно эту деградацию, а не 1920.
9. **Проверять на ширине превью, а не через временный override.** Если, чтобы увидеть «правильную» раскладку, пришлось форсить стили в скриншот-коде (`!important`, отключение media) — это НЕ прошло гейт: пользователь этого override не имеет и видит настоящий рендер файла. Override допустим лишь чтобы разово подтвердить, что широкая раскладка вообще собирается, но итоговая сверка — на том, что файл рендерит сам.

## 1. Разбор макета (интейк)
Для каждого экрана:
1. **`view_image`** макета — рассмотреть внимательно, не по памяти.
2. **Инвентаризация:** выписать все регионы (шапка, табы, тулбар, карточки, таблица, сайдбар, модалка…), все компоненты внутри и все данные (тексты, числа, статусы). Ничего не «додумывать» — если в макете 11 колонок, значит 11.
3. **Маппинг на ДС (компоненты — в приоритете):** прочитать `specs/_cheatsheet.md`, сопоставить каждый регион с компонентом ДС (класс + DOM-сниппет). Правило §0.1: сперва ищем готовый компонент, своя вёрстка — только там, где компонента нет. Для 1–2 компонентов со сложным поведением — полную спеку.
4. **Список пробелов:** чего нет отдельным компонентом (напр. Table, чарт, allocation-бар). По каждому решить: **рисуем на CSS сейчас** (дефолт) / заглушка (внесмасштабное) / стоп-и-спросить (если это крупный недостающий организм и пользователь просил завести его как компонент).

## 2. Метрики с макета (лечит плотность)
До вёрстки снять с картинки и записать (черновиком в комментарии файла или отдельно):
- **Ширина канваса** макета (для этих экранов — обычно **1920**). Верстать под неё.
- **Типошкала:** относительные кегли лейблов / значений / заголовков (в этих макетах лейблы мелкие ~11–12px, значения ~13–14px, тайтлы карточек ~15–16px). Смапить на `--type-*`.
- **Плотность:** паддинги карточек (тут ~16px), зазоры в сетках, высоты строк таблиц/списков, радиусы. Смапить на токены.
- Если ближайший токен заметно расходится с макетом — взять его и записать расхождение в отчёт гейта (не выдумывать промежуточное значение).

## 3. Сборка
- Шаблон — копия `templates/screen/Screen.html`, не с нуля.
- Один линк стилей — `ds.css` (+ `styles/screens.css` для общих раскладок раздела). Свой CSS — **только раскладка** (grid/flex/gap), на токенах.
- Layout — `display:grid/flex` + `gap`, не инлайн-поток и не маржины поэлементно.
- Иконки — только `<i data-icon="имя">` + подключённые `icons-data.js` и `ds-icons.js`. Имена глифов — `specs/Icons.md` (не читать `icons-data.js`). Инлайн-SVG запрещён.
- Классы компонентов брать из чит-шита буквально; **не помнить по памяти** (частая ошибка — `.cbx` вместо `.cb`, `.toast__text` вместо `.toast__msg`). Сомнение — `grep` по нужному `styles/*.css`.
- Статичный мокап: состояния (открытое меню, тултип, выбранная строка) рисуются вёрсткой, без JS-логики.

## 4. Цикл самопроверки (ядро — то, чего не хватило)
Повторять до прохождения гейта:
1. **Открыть на целевой ширине.** Форсить 1920: обёртка фиксированной ширины, либо в снятии скриншота выставить ширину страницы (напр. `document.documentElement.style.width='1920px'`), захват на 100%. Не оценивать в дефолтном узком превью.
2. **Скриншот** своего рендера.
3. **Diff с макетом** — прогнать по рубрике `skills/mockup-fidelity-review.md`, регион за регионом, рядом с `view_image` референса.
4. **Править** расхождения. Особое внимание сеткам с `grid-template`/areas — они чаще всего «разъезжаются» и это видно только на нужной ширине.
5. **Повтор** до **pass**. Только потом — следующий экран.

## 5. Отдача
- `check_design_system` чист (карточки `@dsCard`, если экран в витрине).
- Гейт `skills/mockup-fidelity-review.md` — **pass** с приложенным сравнением.
- Тогда `ready_for_verification({path})`. Встроенный верификатор — бэкап на консоль/ДС/грубый layout, **не** замена сверки с макетом: фиделити гейтишь ты сам до него.
- Новую плотность/приёмы, отлаженные на эталонном экране, задокументировать (обновить `styles/screens.css`, спеку раздела), чтобы следующие экраны наследовали, а не изобретали заново.

## 6. Частые грабли (из реального провала)
- Оценка в узком превью → «на 1920 будет норм» без проверки. **Нет.** Проверяй на 1920.
- Слишком воздушно (крупный кегль + большие зазоры) → «прототип», а не боевой IB-экран. Ужимай по метрикам §2.
- Плитки с 2-колоночным `grid` «подпись→значение» разъезжаются — выверять на ширине.
- Заглушки вместо рисунка — недодача. Рисуй.
- Классы по памяти — ломается тихо. Бери из чит-шита/`grep`.
