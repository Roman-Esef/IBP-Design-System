# Дизайн-система — правила ведения

## Язык
Отвечать и рассуждать в проекте только на русском.

## Формат ДС (ВАЖНО — отличие от исходного проекта)
Эта ДС — **чистый ванильный HTML/CSS**, без React. Источник истины — токены (`styles/*.css`), классы компонентов и HTML-страницы документации. **Не создавать** React-компоненты (`<Имя>.jsx` / `<Имя>.d.ts`): компилятор ДС собирает витрину из карточек `@dsCard` на страницах и реестр токенов из CSS — этого достаточно, и с файлами может работать любой агент/харнес напрямую. Раздел «Для разработчиков» на страницах — framework-agnostic хендофф (redline + псевдокод), а не React-код.

## Жёсткие запреты (НИКОГДА)
- Не читать `icons-data.js` (671 KB, одна строка). Имена глифов — `specs/Icons.md`. Один SVG — через run_script: `(await readFile('icons-data.js')).match(/"имя":"((?:[^"\\]|\\.)*)"/)[1]`.
- Не читать страницу `<Имя>.html` целиком. Сначала спека `specs/<Имя>.md`; мало — `grep` или `read_file` с offset/limit.
- Не перечитывать файл после своей правки «для самопроверки». Проверка одна — в конце через `ready_for_verification`.
- Не инлайнить SVG в разметку экранов — только `<i data-icon="имя"></i>` (см. «Сборка экранов»).
- Не выдумывать цвета/размеры/радиусы/шрифты — только токены `var(--*)` и классы ДС. Нет токена → спроси.
- Не перезаписывать файл через write_file при точечной правке — только `str_replace_edit` (массовые правки — run_script, чтобы содержимое не шло через контекст).
- Не трогать `uploads/` — это референсы пользователя.
- Не писать React-компоненты (`.jsx`/`.d.ts`) и не подключать их — ДС ванильная (см. «Формат ДС»).

## Назначение ДС
Точка истины по визуальному языку продукта — для нас (дизайнеров) и для разработчиков, которые собирают по ней компоненты (см. раздел «Для разработчиков»). В будущем контент страниц соберут в md для автоматического разбора агентом, поэтому текст должен быть точным и однозначным: чёткие заголовки, таблицы вместо прозы, явные значения токенов.

## Версионирование
В шапке страницы (masthead `.meta`) — поля «Версия» и «Обновлено».
- Менялся сам компонент (стили/поведение/токены) → «Версия» +0.1 И «Обновлено» = текущая дата.
- Менялась только страница → только «Обновлено» = текущая дата.
- Формат даты ДД.ММ.ГГГГ. Новые компоненты — с `v1.0`.

## Контракт страницы компонента
Обязательные разделы в этом порядке; имена h2 — контракт для агентов, только эти слова (варианты «Когда использовать», «Текстовка» и т.п. запрещены):
1. Шапка (masthead) с «Версия» и «Обновлено»
2. Конструктор — первым после шапки
3. Использование (h3: когда использовать/не использовать, сравнения с соседями, примеры в интерфейсе)
4. Анатомия
5. Варианты — контейнер всех осей конфигурации (типы, тоны, формы, ориентация, режимы, «С бейджем»…), каждая ось — h3
6. Размеры
7. Контент (текст, переполнение, иконографика)
8. Поведение — контейнер: позиционирование, авто-разворот, анимации, тайминги, адаптивность; каждая тема — h3
9. Состояния
10. Доступность
11. Типографика
12. Цвета
13. Для разработчиков

Правила:
- Отсутствующий раздел — добавить. Неприменимый (например «Типографика» без текста) — оставить с одной строкой «Не применимо: …».
- Уникальные темы — h3 внутри канонических h2 («Варианты», «Поведение», «Использование», «Контент»); новые h2 не вводить.
- Бейджи («Новое», «Расширение») — отдельным элементом внутри h2, не в тексте заголовка.
- Интерактивные блоки (Конструктор/playground) — наверху, сразу после шапки.
- Сегмент-контролы не переносить в 2 ряда: не помещаются — `<select>`.
- Блок «Что предлагаем добавить»/«Бэклог» — вне контракта, в самом конце после «Для разработчиков».

### Раздел «Для разработчиков»
Хендофф-пакет (не копипаст-код под стек), только два блока с copy-to-clipboard:
- **Точные размеры (redline)** — таблица высот/паддингов/радиусов/зазоров/типографики по размерам. Значения — через `getComputedStyle(...)` с реально отрендеренного экземпляра (НЕ `getBoundingClientRect()` — на скрытых off-screen вернёт 0), чтобы не расходились с CSS.
- **Поведение · псевдокод (framework-agnostic)** — сложная логика (позиционирование, измерения, клавнавигация) как алгоритм, не vanilla-JS сниппет.
- «Анатомию DOM», «Рекомендуемый API (React)», «Справочник классов» НЕ добавлять.

### Конструктор (playground)
Панель настроек — на pg-kit (`styles/pg-kit.css` + `pg-kit.js` подключены):
- Между настройками и превью — Splitter из ДС (`.spl`), перетаскивание меняет ширину, двойной клик — сброс. Не убирать.
- Дефолт для всех страниц: панель настроек 360px (`--pg-controls-w` в `pg-kit.css`), превью 1fr, настройки в одну колонку. На странице ничего не прописывать — только подключить pg-kit. Две колонки — лишь если пользователь сам расширил панель сплиттером (≥640px).
- Селекты из 2–4 коротких опций → SegmentControl; 5+ опций или длинные подписи → выпадающий список (сегменты не переносить в 2 ряда). Принудительно оставить селект — `data-pg-keep-select` на `.ctl`.
- Выбор иконки — сетка глифов, не список названий.
- Неактуальные контролы скрывать полностью (`.is-off`), не блокировать/затемнять.

## Порядок компонентов в списках
В каждой группе (Основы / Атомы / Молекулы / Организмы) карточки в `index.html` и пункты в `scripts/ds-nav.js` — строго по алфавиту видимого названия (`card__title` / `label`): «Основы» по кириллице (Иконки, Иллюстрации, Скругления, Тени, Типографика, Цвета), остальные по латинице (имя = имя файла). Заглушки `soon: true` — в конце группы, вне алфавита. Новый компонент вставлять на алфавитно верную позицию, не в конец.

## Экономия токенов (приоритет над привычками)
1. **Спеки вместо страниц.** У каждого компонента `specs/<Имя>.md` (6–9 KB вместо 40–70 KB страницы): YAML-шапка, правила, DOM-анатомия, классы. Манифест — `specs/_index.md`. Для молекул/организмов читай спеки вложенных компонентов, не страницы.
2. **Спеки держать актуальными.** Изменил/создал компонент → тем же ходом обнови/создай спеку (образец `specs/_TEMPLATE.md`). Устаревшая спека хуже отсутствующей.
3. **Точечно.** `grep` вместо чтения целиком; `str_replace_edit` вместо перезаписи; массовые правки — run_script.
4. **Новые страницы** — копией ближайшей (copy_files) + точечные правки, не boilerplate. В правильную папку: `pages/{foundations,atoms,molecules,organisms}/`.
5. **Скриншоты** в `screenshots/` — временные, не плодить, чистить. Несколько состояний — батчить (multi_screenshot / один eval_js с объектом), не серией.
6. **Полнота index.html и ds-nav.js — при КАЖДОМ новом компоненте.** Оба обязаны перечислять ВСЕ файлы из `pages/*/`. Сверка без чтения страниц: `list_files pages/<категория>/` → сравнить с `ds-nav.js` (читается целиком) → `grep` по `index.html` на `<Имя>.html`. Расхождение — добавить в оба места.

## Сборка экранов и организмов
1. **Контекст = один файл.** Читай `specs/_cheatsheet.md` (~18 KB) — все компоненты: css, deps, назначение, DOM-сниппет. Полную спеку — только для 1–2 компонентов со сложным поведением. CSS компонентов не читать.
2. **Один линк стилей** — только `ds.css` (импортирует всю ДС), не styles/* поштучно. Новый компонент → добавить его @import и в `ds.css`, и в корневой `styles.css`.
3. **Шаблон** — копия `templates/screen/Screen.html`, не с нуля.
4. **Иконки** — только `<i data-icon="имя"></i>` + подключённые `scripts/icons-data.js` и `scripts/ds-icons.js` (сам вставляет SVG в currentColor; размер — width/height, дефолт 24px). Инлайн-SVG запрещён.
5. **Стили экрана** — токены и классы ДС; свой CSS только раскладка (grid/flex/gap).
6. **Нет нужного компонента в ДС** (Input, Table, Modal, Card…) — не выдумывать на ходу. Остановись, назови недостающее, сначала создаём его как компонент ДС, потом экран.
7. **Экран статичен** (мокап): состояния показывать вёрсткой (открытый дропдаун — отрисован открытым). JS — только если явно попросили кликабельный прототип.
8. **Чит-шит актуален**: новый/изменённый компонент → обнови его блок в `specs/_cheatsheet.md` тем же ходом.

### Чек-лист: новый экран
1. Прочитай `specs/_cheatsheet.md`, составь список нужных компонентов.
2. Чего-то нет в ДС — остановись, сообщи, дождись решения.
3. Для 1–2 сложных компонентов прочитай полные спеки. Больше ничего.
4. Скопируй `templates/screen/Screen.html` под именем экрана.
5. Собери разметку из DOM-сниппетов; иконки — `<i data-icon>`; свой CSS — только раскладка.
6. `ready_for_verification`.

## Новый компонент
Промпт-роль (действует автоматически при референсе + названии компонента): ты топовый дизайнер интерфейсов и дизайн-систем. Собери страницу компонента и добавь в ДС на токенах и шрифтах ДС (ванильный HTML/CSS, без React). Не знаешь, что брать — спрашивай. Предложи, чего не хватает (состояния, корнер-кейсы). Основной референс — Material Design, но бери лучшие практики и из других ДС. Делай по аналогии с соседними компонентами.

### Чек-лист (строго по порядку)
1. Прочитай `specs/_cheatsheet.md` и спеки 1–2 похожих. Страницы целиком не читай.
2. Создай `styles/<имя>.css` — на токенах ДС.
3. Добавь `@import url("styles/<имя>.css");` и в `ds.css`, и в корневой `styles.css` (в конец группы).
4. Страница `<Имя>.html`: копия ближайшего похожего (copy_files) в нужную папку + точечные правки. Шапка: «v1.0», текущая дата. Разделы — по контракту выше. Первой строкой — карточка `<!-- @dsCard group="..." ... -->`.
5. Создай спеку `specs/<Имя>.md` (образец `specs/_TEMPLATE.md`).
6. Добавь блок компонента в `specs/_cheatsheet.md`.
7. Добавь карточку в `index.html`: `grep` соседних карточек группы → `str_replace_edit` на алфавитную позицию, обнови счётчик в `.sec-head`.
8. Добавь пункт в `scripts/ds-nav.js` — та же группа, алфавитная позиция.
9. Сверься: число файлов в `pages/<категория>/` = число карточек в index.html = пунктов в ds-nav.js.
10. `check_design_system` чист → `ready_for_verification`.

## Правка существующего компонента
1. Прочитай `specs/<Имя>.md`; страницу — точечно (grep/offset).
2. Правь через `str_replace_edit`.
3. Версия/дата по правилу «Версионирование»; обнови те же поля в YAML-шапке спеки.
4. Обнови изменённое в `specs/<Имя>.md` и в блоке `specs/_cheatsheet.md`.
5. `ready_for_verification`.

## Общие принципы работы
Очевидные баги правь сам. По неоднозначному (новый раздел, смена визуала, изменение API) — сначала спроси. Не переписывай разделы без замечаний.

## Тона состояний Button и Link
У Button и Link есть тоновые классы `.btn--<tone>` и `.link--<tone>` (info/warning/error/success). Кнопки и ссылки внутри Alert и SnackBar красятся именно этими классами (совпадают с тоном плашки); локальная подмена `--primary`/`--link-fg` убрана (17.07.2026). Бывшее предупреждение компилятора о 37 токенах Snack/Alert больше не возникает — `check_design_system` чист.

## Структура проекта
- `index.html` — обзор ДС (карточки), в корне.
- `styles.css` — **корневая точка входа компилятора ДС** (@import всех токенов и стилей); её линкуют потребители. Зеркало `ds.css`.
- `ds.css` — единая точка подключения ДС для экранов (@import всех styles/*), в корне; для страниц-экранов.
- `templates/screen/Screen.html` — стартовый шаблон экрана.
- `thumbnail.html` — иконка ДС для домашней витрины.
- `pages/` — документация компонентов по категориям (первой строкой каждой — карточка `@dsCard`):
  - `foundations/` — Typography, Colors, Icons, Illustrations, Radius, Elevation.
  - `atoms/` — Buttons, IconButton, Link, LabelHelper (Label + Helper), Checkbox, Radiobutton, Switch, Badge, Chip, Avatar, Divider, ProgressBar.
  - `molecules/` — Alert, ButtonGroup, Breadcrumbs, Tooltip, ContextMenu, Tab, SegmentControl, Splitter, InputText, InputDate, InputDateRange, InputAmountRange, InputAutocomplete, Select, ReadOnlyField, Toast, Pagination, NavTile.
  - `organisms/` — Entity, Modal, NavPanel, PageHeader, Popover, RiskMetric, SnackBar, TableCell, TableFilter, Tile.
  - `patterns/` — SectionsAudit.
  - Внутри `pages/<категория>/`: стили `../../styles/*.css`, скрипты `../../scripts/*.js`, ссылки на страницы `../<категория>/<Имя>.html`, на index `../../index.html`. Компоненты подкачки (Alert, Label + Helper) переехали в molecules/atoms — отдельной папки `loading/` больше нет.
- `scripts/` — общие (`ds-icons.js`, `ds-nav.js`, `ds-toc.js`, `pg-kit.js`, `icons-data.js`, `image-slot.js`) и per-компонентные `<имя>.page.js`.
  - `ds-icons.js` — `<i data-icon="имя"></i>` → SVG из icons-data.js в currentColor.
  - `ds-nav.js` — левая навигация; полный список страниц путями от корня, префикс через `window.__DS_ROOT` (страницы в `pages/*/` объявляют `'../../'`; index.html — по умолчанию `''`). В начале `mount()` — no-op guard: монтируется только при наличии `<main class="page">` (на карточках `@dsCard`, тянущих скрипт через `_ds_bundle.js`, молча выходит).
  - `ds-toc.js` — правый оглавитель; тот же no-op guard.
  - `pg-kit.js` — слой панелей «Конструктор» (см. раздел «Конструктор»).
- `styles/` — CSS компонентов и фундамента (плоско): `typography.css` (SB Sans Display/Text/Screen, токены `--type-*`, классы `.ds-*`), `colors.css` (примитивы `--c-*` + семантические Static/Active/Situative/Local), по файлу на компонент.
- `specs/` — md-спеки с YAML-шапкой: `_index.md` (манифест), `<Имя>.md` (поле `page:` = путь `pages/<категория>/<Имя>.html`), `Icons.md` (все глифы).
