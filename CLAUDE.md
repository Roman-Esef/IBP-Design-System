# Дизайн-система — правила ведения

## Язык
Отвечать и рассуждать в этом проекте только на русском языке.

## Жёсткие запреты (НИКОГДА, без исключений)
- НИКОГДА не читать `icons-data.js` (671 KB, одна строка — сжигает контекст). Имена глифов — `specs/Icons.md`.
- НИКОГДА не читать страницу `<Имя>.html` целиком. Сначала спека `specs/<Имя>.md`; если мало — `grep` по странице или `read_file` с offset/limit.
- НИКОГДА не перечитывать файл после своей правки «для самопроверки». Проверка — один раз в конце через `ready_for_verification`.
- НИКОГДА не инлайнить SVG в разметку экранов — только `<i data-icon="имя"></i>` (см. «Сборка экранов»).
- НИКОГДА не выдумывать цвета, размеры, радиусы, шрифты — только токены `var(--*)` и классы ДС. Нет подходящего токена → спроси.
- НИКОГДА не переписывать существующий файл через write_file, если правка точечная — использовать `str_replace_edit`.
- НИКОГДА не трогать `uploads/` — это референсы пользователя.

## Версионирование компонентов
Каждая страница компонента имеет в шапке (masthead `.meta`) поля «Версия» и «Обновлено».
- **Версия** (например `v1.0`) — повышать при изменении самого компонента (его стилей/поведения/токенов).
- **Обновлено** — ставить ТЕКУЩУЮ дату в формате ДД.ММ.ГГГГ при ЛЮБОМ изменении страницы/компонента.
- Новые компоненты начинаем с `v1.0` и текущей даты.

## Структура страниц компонентов
- Интерактивные блоки (например «Конструктор»/playground) размещаем НАВЕРХУ страницы, самыми первыми — сразу после шапки.
- Сегмент-контролы не должны переноситься в 2 ряда: если опции не помещаются в один ряд — использовать выпадающий список (`<select>`), а не перенос.
- Каждая страница компонента должна включать раздел **«Для разработчиков»** (ближе к концу страницы, обычно после Accessibility/Colors). Наши разработчики собирают компоненты на React, поэтому раздел не должен быть «код на копипаст» под конкретный стек — это хендофф-пакет, из которого разработчик сам берёт нужное для вёрстки:
  - **Точные размеры (redline)** — таблица высот/паддингов/радиусов/зазоров/типографики по размерам компонента. Значения — НЕ хардкод, а измеряются через `getComputedStyle(...)` (не `getBoundingClientRect()` — на скрытых off-screen элементах он возвращает 0) на реально отрендеренном экземпляре компонента, чтобы таблица не расходилась с CSS при будущих правках.
  - **Поведение · псевдокод (framework-agnostic)** — сложная логика (позиционирование индикаторов, измерение размеров, клавиатурная навигация) описывается как framework-agnostic псевдокод/алгоритм, а не как vanilla-JS сниппет для вставки.
  - Больше ничего в этом разделе не нужно: «Анатомию DOM», «Рекомендуемый API компонента (React)» и «Справочник классов и атрибутов» не добавлять — ни на новых страницах, ни при правках существующих.
  - Copy-to-clipboard кнопки на всех блоках кода.

## Назначение этой ДС (учитывать при любой работе)
- ДС хранит информацию по компонентам, стилям и токенам для нас как для дизайнеров — точка истины по визуальному языку продукта.
- ДС нужна разработчикам, чтобы по ней собирать компоненты на React (см. правило про раздел «Для разработчиков» выше).
- В будущем весь контент страниц будет собран в md-файлы, по которым агент будет искать информацию и отвечать на вопросы по ДС. Поэтому текст страниц должен быть точным, структурированным и однозначно читаемым не только человеком, но и при последующем автоматическом разборе (чёткие заголовки разделов, таблицы вместо расплывчатых описаний, явные значения токенов).

## Порядок компонентов в списках
Внутри каждой группы (Основы / Атомы / Молекулы / Организмы / Компоненты подкачки) карточки в `index.html` и пункты в `scripts/ds-nav.js` идут в строгом алфавитном порядке по видимому названию (`card__title` / `label`) — для «Основы» это кириллица (Иконки, Скругления, Тени, Типографика, Цвета), для остальных групп — латиница (имя компонента = имя файла страницы). Нереализованные пункты-заглушки (`soon: true`) остаются в конце своей группы, вне алфавитного счёта. При добавлении нового компонента (Шаги 7–8 чек-листа «новый компонент» ниже) вставляй карточку/пункт на алфавитно верную позицию, а не в конец списка.


Когда я пишу «обнови и проверь» (или «проверь и обнови») без уточнений — проведи полный аудит ТЕКУЩЕЙ открытой страницы компонента по шагам ниже. Ничего не спрашивай перед началом. Выполняй шаги строго по порядку, ни один не пропускай.

**Шаг 1. Работоспособность.**
Открой страницу, прочитай логи консоли (`get_webview_logs`). Прощёлкай интерактив: каждый контрол конструктора, кнопки копирования кода, переключатели состояний. Ошибки в консоли и сломанный интерактив — чинить сразу.

**Шаг 2. Обязательные разделы (единый контракт h2).** Имена h2 — контракт для агентов: только эти слова, без вариаций («Когда использовать», «Текстовка» и т.п. запрещены). Проверь, что на странице есть ВСЕ разделы, в этом порядке:
1. Шапка (masthead) с полями «Версия» и «Обновлено»
2. Конструктор — первый раздел после шапки
3. Использование (сюда же h3-подразделами: когда использовать/не использовать, сравнения с соседними компонентами, примеры в интерфейсе)
4. Анатомия
5. Варианты — контейнер для всех осей конфигурации (типы, тоны, формы, ориентация, режимы, «С бейджем» и т.п.), каждая ось — h3
6. Размеры
7. Контент (правила текста, переполнение, иконографика)
8. Поведение — контейнер для позиционирования, авто-разворота, анимации, таймингов, адаптивности; каждая тема — h3
9. Состояния
10. Доступность
11. Типографика
12. Цвета
13. «Для разработчиков» (redline-таблица + поведение · псевдокод, кнопки copy на всех блоках кода). Только эти два блока — «Анатомию DOM», «Рекомендуемый API компонента (React)» и «Справочник классов и атрибутов» не добавлять.

Правила контракта:
- Отсутствующий раздел — добавить, а не отметить как «нет». Если раздел неприменим (например «Типографика» у компонента без текста) — раздел всё равно существует и содержит одну строку «Не применимо: …».
- Уникальные для компонента темы живут h3-подразделами внутри канонических h2 («Варианты», «Поведение», «Использование», «Контент») — новые h2 не вводить.
- Бейджи («Новое», «Расширение») — отдельным элементом внутри h2, не частью текста заголовка.
- Блок «Что предлагаем добавить»/«Бэклог» — вне контракта: отдельный блок в самом конце страницы, после «Для разработчиков».

**Шаг 3. Конструктор.** Убедись, что панель настроек собрана на pg-kit (`styles/pg-kit.css` + `pg-kit.js` подключены). Проверь каждое из условий:
- Между настройками и превью стоит Splitter из ДС (`.spl`), ширина панелей регулируется перетаскиванием; двойной клик — сброс. Сплиттер никогда не убирать.
- Панель настроек по умолчанию — **360px** (`--pg-controls-w`, задаётся в `styles/pg-kit.css`), превью — гибкое (1fr). Это дефолт для ВСЕХ страниц; отдельно на странице ничего не прописывать — просто подключить `pg-kit.css`/`pg-kit.js`. Сплиттер позволяет растянуть панель настроек при необходимости — это не меняет дефолт.
- Настройки по умолчанию в ОДНУ колонку. Две колонки — только если пользователь сам расширил панель сплиттером (≥640px). Не менять это соотношение.
- Селекты из 2–4 коротких опций отображаются как SegmentControl из ДС; селекты из 5+ опций или с длинными подписями остаются выпадающими списками (сегменты НИКОГДА не переносятся в 2 ряда). Чтобы принудительно оставить селект — атрибут `data-pg-keep-select` на `.ctl`.
- Выбор иконки — сетка глифов, не список названий.
- Неактуальные в текущей конфигурации контролы скрыты полностью (`.is-off`), а не заблокированы/затемнены.

**Шаг 4. Redline-значения.** Значения в таблицах раздела «Для разработчиков» должны читаться через `getComputedStyle(...)` с реально отрендеренного экземпляра (НЕ `getBoundingClientRect()` на скрытых элементах — вернёт 0) — либо скриптом на странице, либо сверь вручную с CSS. Расхождение таблицы с CSS — это баг, исправить.

**Шаг 5. Текст.** Заголовки разделов чёткие, значения токенов и размеров даны явно (таблицы, не проза), формулировки однозначны для будущего автоматического разбора в md. Противоречий между разделами нет (например, «Цвета» совпадает с реальным CSS).

**Шаг 6. Версия и дата.**
- Менялся сам компонент (стили/поведение/токены) → поднять «Версия» на 0.1 И поставить текущую дату в «Обновлено».
- Менялась только страница → «Версия» не трогать, дату «Обновлено» поставить текущую (формат ДД.ММ.ГГГГ).
- Ничего не менялось → ничего не трогать.

**Шаг 7. Финал.**
Очевидные баги и несоответствия исправляй сам, без вопросов. По неоднозначным решениям (новый раздел, смена визуала, изменение API) — сначала спроси. В конце дай краткий отчёт списком: что проверено ✓, что исправлено, что требует моего решения. Не переписывай разделы, к которым нет замечаний.

## Экономия токенов (обязательно, приоритет над привычками)
Контекст дорог — лимит сгорает на чтении больших страниц. Правила:

1. **Спеки вместо страниц.** У каждого компонента есть `specs/<Имя>.md` (6–9 KB вместо 40–70 KB страницы): YAML-шапка (css, deps, версия), правила из всех разделов, DOM-анатомия, справочник классов. Манифест — `specs/_index.md`. При сборке молекул/организмов читай спеки вложенных компонентов — страницы `<Имя>.html` целиком НЕ читать. Если спеки не хватило — точечный `grep` по странице или `read_file` с offset/limit, не целиком.
2. **Спеки держать в актуальном состоянии.** Изменил компонент (CSS/анатомию/поведение/версию) → тем же ходом обнови его спеку. Создал компонент → сразу создай спеку по образцу `specs/_TEMPLATE.md`. Устаревшая спека хуже её отсутствия.
3. **`icons-data.js` (671 KB, одна строка) никогда не читать.** Имена всех глифов — в `specs/Icons.md`. Один SVG доставать через run_script: `(await readFile('icons-data.js')).match(/"имя":"((?:[^"\\]|\\.)*)"/)[1]` — контент не попадёт в контекст.
4. **Точечные операции.** `grep` вместо чтения файла целиком; `str_replace_edit` вместо перезаписи; массовые правки — через run_script (содержимое файлов не проходит через контекст). Свою страницу после правки не перечитывать для самопроверки — проверка один раз в конце через верификатор.
5. **Новые страницы** делать копией ближайшей существующей (copy_files) с последующими точечными правками, а не писать boilerplate заново. Класть в правильную папку категории: `pages/atoms/`, `pages/molecules/`, `pages/organisms/`, `pages/loading/` или `pages/foundations/`.
6. **Скриншоты** в `screenshots/` — временные артефакты; не плодить, папку периодически чистить. Проверки нескольких состояний батчить в один вызов (multi_screenshot / один eval_js с объектом ответов), а не серией одиночных.
7. **Карточку нового компонента в `index.html`** добавлять точечно: найти якорь через `grep` (последняя карточка нужной группы) и вставить `str_replace_edit`-ом; файл целиком не читать.
8. **Полнота index.html и ds-nav.js — проверять при КАЖДОМ новом компоненте.** `index.html` разбит на секции Атомы/Молекулы/Организмы (+ Основы, Компоненты подкачки); `scripts/ds-nav.js` — на те же группы в левом меню. Оба места обязаны перечислять ВСЕ файлы из `pages/atoms/`, `pages/molecules/`, `pages/organisms/`, `pages/loading/`, `pages/foundations/` без исключений. Быстрая сверка без чтения страниц целиком: `list_files` по `pages/<категория>/` → сравнить со списком в `ds-nav.js` (читается целиком, он маленький) → `grep` по `index.html` на имя `<Имя>.html`, чтобы убедиться, что карточка существует. Расхождение — добавить недостающее в оба места, никогда не оставлять только в одном.

## Сборка экранов и организмов на ДС
Экраны продукта собираем максимально экономно:

1. **Контекст = один файл.** Сначала читай `specs/_cheatsheet.md` (~18 KB) — там все компоненты: css, deps, назначение, DOM-сниппет. Полную спеку `specs/<Имя>.md` брать только для 1–2 компонентов со сложным поведением (Pagination, DropdownList…). CSS компонентов не читать вовсе — классы и DOM уже в спеках.
2. **Один линк стилей.** На экране подключать только `ds.css` (импортирует всю ДС), не перечислять styles/* поштучно. Новый компонент → добавить его @import в ds.css.
3. **Шаблон.** Новый экран — копия `_template/Screen.html` (copy_files) в корень проекта, не с нуля.
4. **Иконки не инлайнить.** На экранах — `<i data-icon="имя"></i>` + подключённые `scripts/icons-data.js` и `scripts/ds-icons.js` (хелпер сам вставляет SVG и красит в currentColor; размер — width/height элемента, дефолт 24px). Инлайн-SVG в разметке экрана запрещён — это тысячи лишних токенов вывода.
5. **Стили экрана** — только токены (`var(--*)`) и готовые классы ДС; свой CSS — только раскладка (grid/flex/gap).
6. **Нет нужного компонента в ДС (например Input, Table, Modal, Card) — НЕ выдумывать его на ходу внутри экрана.** Остановись и скажи пользователю: каких компонентов не хватает — сначала создаём их как полноценные компоненты ДС (чек-лист ниже), потом собираем экран.
7. **Интерактив на экранах** — по умолчанию экран статичен (мокап): состояния показывать вёрсткой (открытый дропдаун — просто отрисован открытым). JS-логику писать только если пользователь явно попросил кликабельный прототип.
8. **Чит-шит держать актуальным**: новый/изменённый компонент → обновить его блок в `specs/_cheatsheet.md` тем же ходом, что и спеку.

### Чек-лист: новый экран (строго по порядку)
1. Прочитай `specs/_cheatsheet.md`. Составь список нужных компонентов экрана.
2. Если чего-то в ДС нет — остановись, сообщи список недостающего, дождись решения.
3. Для 1–2 компонентов со сложным поведением прочитай полные спеки `specs/<Имя>.md`. Больше ничего не читай.
4. Скопируй `_template/Screen.html` в корень проекта под именем экрана (например `Screen_СписокСделок.html`).
5. Собери разметку из DOM-сниппетов чит-шита. Иконки — только `<i data-icon="имя"></i>` (имена — specs/Icons.md). Свой CSS — только раскладка.
6. `ready_for_verification`. Готово.

## Роль и подход при создании нового компонента
Когда пользователь прикладывает референсы и называет компонент — этот промпт действует автоматически, отдельно его писать не нужно:

Ты топовый дизайнер интерфейсов и дизайн-систем. Собери страницу компонента и добавь её в дизайн-систему. Используй токены и шрифты дизайн-системы. Если не знаешь, что брать — спрашивай. Посмотри, чего не хватает в документации, и предложи, что можно добавить (отсутствующие состояния, корнер-кейсы и т.д.). Ориентируйся на Material Design как основной референс, но не ограничивайся им — бери лучшие практики из других дизайн-систем тоже. Сделай по аналогии с другими компонентами.

Механика выполнения — чек-лист «новый компонент» ниже.

## Чек-лист: новый компонент (строго по порядку, ни один пункт не пропускать)
1. Прочитай `specs/_cheatsheet.md` и спеки 1–2 похожих компонентов. Страницы целиком не читай.
2. Создай `styles/<имя>.css` — только на токенах ДС (`var(--*)`).
3. Добавь `@import url("styles/<имя>.css");` в `ds.css` (в конец соответствующей группы).
4. Страница `<Имя>.html`: скопируй ближайший похожий компонент через copy_files в нужную папку — `pages/atoms/`, `pages/molecules/`, `pages/organisms/` или `pages/loading/` — и правь точечно. В шапке: «Версия v1.0», «Обновлено» = текущая дата. Состав и порядок разделов — как в Шаге 2 команды «обнови и проверь».
5. Создай спеку `specs/<Имя>.md` по образцу `specs/_TEMPLATE.md`.
6. Добавь блок компонента в `specs/_cheatsheet.md` (формат — как у соседних блоков).
7. Добавь карточку в `index.html`: найди `grep`-ом соседние карточки нужной группы (Атомы/Молекулы/Организмы), вставь новую `str_replace_edit`-ом на алфавитно верную позицию (см. «Порядок компонентов в списках»), обнови счётчик карточек в `.sec-head` этой секции. Файл целиком не читай.
8. Добавь пункт в `scripts/ds-nav.js` — в ту же группу (Атомы/Молекулы/Организмы), на алфавитно верную позицию, тем же `str_replace_edit`-ом.
9. Сверься: `list_files` по `pages/<категория>/` — количество файлов должно совпадать с числом карточек в `index.html` и пунктов в `ds-nav.js` для этой группы.
10. Вызови `ready_for_verification` на новой странице. Готово.

## Чек-лист: правка существующего компонента
1. Прочитай `specs/<Имя>.md`. Страницу — только точечно (grep/offset) по месту правки.
2. Правь CSS/страницу через `str_replace_edit`.
3. Менялся сам компонент (стили/поведение/токены) → «Версия» +0.1 И дата; менялась только страница → только дата. Обнови те же поля в YAML-шапке спеки.
4. Обнови изменённые места в `specs/<Имя>.md` и в блоке компонента в `specs/_cheatsheet.md`.
5. `ready_for_verification`. Готово.

## Структура проекта
- `index.html` — обзорная страница ДС (карточки всех компонентов), остаётся в корне.
- `ds.css` — единая точка подключения всей ДС для экранов (@import всех styles/*), в корне.
- `_template/Screen.html` — стартовый шаблон экрана, в корне.
- `pages/` — все страницы-документация компонентов, разложены по атомарным категориям:
  - `pages/foundations/` — Typography, Colors, Icons, Radius, Elevation.
  - `pages/atoms/` — Buttons, IconButton, Link, Checkbox, Radiobutton, Switch, Badge, Chip, Avatar, Divider, ProgressBar.
  - `pages/molecules/` — ButtonGroup, Breadcrumbs, Tooltip, ContextMenu, Tab, SegmentControl, Splitter, InputAutocomplete, Select, ReadOnlyField, Toast, Pagination.
  - `pages/organisms/` — Modal, Popover.
  - `pages/loading/` («компоненты подкачки») — LabelHelper.
  - Каждая страница внутри `pages/<категория>/` подключает стили как `../../styles/*.css` и скрипты как `../../scripts/*.js`; ссылки на другие страницы — вида `../<категория>/<Имя>.html`; на index — `../../index.html`.
- `scripts/` — все JS-файлы: общие (`ds-icons.js`, `ds-nav.js`, `ds-toc.js`, `pg-kit.js`, `icons-data.js`, `image-slot.js`) и per-компонентные `<имя>.page.js` (конструкторы страниц).
  - `ds-icons.js` — хелпер иконок для экранов: `<i data-icon="имя"></i>` → SVG из icons-data.js, перекрашенный в currentColor.
  - `ds-nav.js` — левая навигация; хранит полный список страниц с путями от корня (`pages/<категория>/<Имя>.html`) и сам достраивает префикс через `window.__DS_ROOT` (страницы в `pages/*/` объявляют `window.__DS_ROOT = '../../'` перед подключением скрипта; index.html — без объявления, по умолчанию `''`).
  - `pg-kit.js` — общий слой панелей «Конструктор»: раскладка настройки-слева/превью-справа со Splitter из ДС, автозамена селектов из 2–4 коротких опций на SegmentControl, селектов-иконок — на сетку глифов, скрытие неактуальных контролов (`.ctl.is-off`). Подключать на каждой новой странице с конструктором; чтобы оставить конкретный селект селектом — атрибут `data-pg-keep-select` на `.ctl`.
- `styles/` — CSS компонентов и фундамента, без изменений (плоская структура): `styles/typography.css` — типографика (SB Sans Display/Text/Screen), токены `--type-*` и классы `.ds-*`; `styles/colors.css` — цвета: примитивы `--c-*` + семантические токены (Static/Active/Situative/Local); `styles/button.css`, `styles/chip.css`, `styles/segment-control.css` и т.д. — по компоненту.
- `specs/` — md-спеки компонентов с YAML-шапкой (см. «Экономия токенов»): `specs/_index.md` — манифест, `specs/<Имя>.md` — по одной на страницу (поле `page:` указывает путь вида `pages/<категория>/<Имя>.html`), `specs/Icons.md` — список всех глифов.

Каждая страница подключает нужные `styles/*.css` (с учётом глубины пути) и использует токены, а не хардкод значений.
